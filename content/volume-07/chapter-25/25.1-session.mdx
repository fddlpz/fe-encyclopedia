## 章节定位

会话管理 不再只按“是什么、怎么用”展开，而是放回「第 25 章 身份认证与授权」的知识链路中理解：区分认证、授权和会话管理，设计凭证生命周期、撤销机制和跨端安全边界。

> **本节要解决的真实问题**：当项目里遇到 会话管理、Session、Cookie、会话 相关设计、选型、接入或排障时，能说清楚边界、取舍、验证方式和失败后的回退策略。

### 学习目标

| 层次 | 学完后应该能做到 |
|---|---|
| 概念定位 | 解释 会话管理 在 身份认证与授权 中解决的真实问题 |
| 机制理解 | 拆解 Session、Cookie 与失败路径 |
| 工程判断 | 为 会话 选择可验证、可回退的落地方案 |
| 验证闭环 | 围绕“凭证存在哪里”建立判断标准 |

## 知识地图

认证回答你是谁，授权回答你能做什么，会话负责在多次请求之间维持可信状态。

| 层级 | 本节关注点 | 判断证据 |
|---|---|---|
| 问题入口 | 会话管理、Session、Cookie 会在什么业务场景中暴露 | 管理后台、开放平台 |
| 运行机制 | 凭证存在哪里；如何刷新和撤销；权限变更何时生效 | Set-Cookie 属性、token 过期路径 |
| 工程约束 | Cookie/Token、刷新策略、SSO 协议、MFA | 兼容性、性能、安全、维护成本和回退方案 |
| 关联知识 | 与同章和跨章内容形成可追踪链路 | 关系链接、示例代码、测试和线上数据 |

## 核心机制

这一节建议沿着“术语 -> 机制 -> 证据”的顺序阅读。先不要急着背 API，先确认每个概念在系统里承担什么职责，以及它失败时会出现什么现象。

| 主题 | 需要理解的机制 | 可观察证据 |
|---|---|---|
| 会话管理 | 认证、授权和会话管理分别处理身份确认、资源权限和跨请求可信状态。 | 覆盖登录、刷新、退出、权限变更、设备丢失和跨站请求路径。 |
| Session | 围绕“如何刷新和撤销”拆解概念边界、运行机制和失败路径。 | token 过期路径 |
| Cookie | 围绕“权限变更何时生效”拆解概念边界、运行机制和失败路径。 | 重放测试 |
| 会话 | 围绕“跨站请求如何防护”拆解概念边界、运行机制和失败路径。 | 权限审计日志 |

## 工程决策

会话管理 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| Cookie/Token | 凭证存储、刷新、撤销、设备绑定和风险校验必须一起设计。 | 覆盖登录、刷新、退出、权限变更、设备丢失和跨站请求路径。 |
| 刷新策略 | 在“开放平台”场景下判断收益、成本、失败路径和回退策略。 | token 过期路径 |
| SSO 协议 | 在“多端登录”场景下判断收益、成本、失败路径和回退策略。 | 重放测试 |
| MFA | 在“高风险操作”场景下判断收益、成本、失败路径和回退策略。 | 权限审计日志 |

## 实战路径

推荐按四步推进：

1. **界定场景**：明确当前是新功能接入、旧系统治理、性能优化、安全加固还是线上排障。
2. **建立样例**：用最小代码或配置复现 会话管理 的核心行为，不直接在完整业务里猜测。
3. **接入约束**：把 Cookie/Token、刷新策略、SSO 协议 写进方案，说明不满足时如何降级。
4. **验证沉淀**：用 Set-Cookie 属性、token 过期路径、重放测试 形成证据，并把结论回写到团队文档。

```ts
type ChapterDecision = {
  objective: string
  constraints: string[]
  signals: string[]
  rollback: string
}

/**
 * 生成本节主题的落地决策记录。
 * @param owner 负责验证和维护该方案的角色或小组。
 * @returns 可写入技术方案或评审记录的决策对象。
 */
function createDecisionRecord(owner: string): ChapterDecision {
  return {
    objective: "会话管理",
    constraints: ["会话管理","Session","Cookie","会话"],
    signals: ["Set-Cookie 属性","token 过期路径","重放测试"],
    // 中文注释：回退策略必须在接入前定义，避免上线后只能被动救火。
    rollback: `由 ${owner} 负责准备降级开关、兼容方案和验证清单`,
  }
}

console.table(createDecisionRecord("frontend"))
```

## 诊断与验证

| 现象 | 定位动作 | 判断完成的证据 |
|---|---|---|
| 行为与预期不一致 | 回到最小样例，逐步加入真实数据、环境和配置 | 能明确指出差异来自输入、运行时、配置还是依赖 |
| 性能或稳定性退化 | 记录基线，再观察 Set-Cookie 属性、token 过期路径、重放测试 | 指标回到预算内，并且关键路径有回归测试 |
| 团队理解不一致 | 把术语、边界、反例和回退策略写进评审记录 | 新成员可以按同一清单复现和验证 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 把 JWT 当作天然可撤销会话，会让权限变更和退出登录失效。 | 先用最小场景复现，再决定是否升级方案。 |
| 把 JWT 当作可撤销会话 | 先用最小场景复现，再决定是否升级方案。 |
| 长期保存敏感令牌 | 先用最小场景复现，再决定是否升级方案。 |
| 没有处理退出登录和权限变更 | 先用最小场景复现，再决定是否升级方案。 |
