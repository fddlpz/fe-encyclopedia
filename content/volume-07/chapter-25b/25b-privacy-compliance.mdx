## 工程决策

隐私合规与数据治理 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| 数据分类 | 先定义数据分类和威胁模型，再选择算法、采集策略或保留周期。 | 用测试向量、同意日志、数据删除演练和第三方 SDK 清单验证。 |
| 同意管理 | 在“广告归因”场景下判断收益、成本、失败路径和回退策略。 | 埋点审计 |
| 最小化采集 | 在“日志采集”场景下判断收益、成本、失败路径和回退策略。 | 同意日志 |
| 保留周期 | 在“跨境产品”场景下判断收益、成本、失败路径和回退策略。 | 删除请求演练 |

## 实战路径

推荐按四步推进：

1. **界定场景**：明确当前是新功能接入、旧系统治理、性能优化、安全加固还是线上排障。
2. **建立样例**：用最小代码或配置复现 隐私合规与数据治理 的核心行为，不直接在完整业务里猜测。
3. **接入约束**：把 数据分类、同意管理、最小化采集 写进方案，说明不满足时如何降级。
4. **验证沉淀**：用 数据地图、埋点审计、同意日志 形成证据，并把结论回写到团队文档。

```ts
type ChapterDecision = {
  objective: string
  constraints: string[]
  signals: string[]
  rollback: string
}

/**
 * 生成本节主题的落地决策记录。
 * @param owner 负责验证和维护该方案的角色或小组。
 * @returns 可写入技术方案或评审记录的决策对象。
 */
function createDecisionRecord(owner: string): ChapterDecision {
  return {
    objective: "隐私合规与数据治理",
    constraints: ["隐私合规与数据治理","GDPR","隐私","合规"],
    signals: ["数据地图","埋点审计","同意日志"],
    // 中文注释：回退策略必须在接入前定义，避免上线后只能被动救火。
    rollback: `由 ${owner} 负责准备降级开关、兼容方案和验证清单`,
  }
}

console.table(createDecisionRecord("frontend"))
```

## 诊断与验证

| 现象 | 定位动作 | 判断完成的证据 |
|---|---|---|
| 行为与预期不一致 | 回到最小样例，逐步加入真实数据、环境和配置 | 能明确指出差异来自输入、运行时、配置还是依赖 |
| 性能或稳定性退化 | 记录基线，再观察 数据地图、埋点审计、同意日志 | 指标回到预算内，并且关键路径有回归测试 |
| 团队理解不一致 | 把术语、边界、反例和回退策略写进评审记录 | 新成员可以按同一清单复现和验证 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 前端硬编码密钥或无目的采集数据，都会让治理失去意义。 | 先用最小场景复现，再决定是否升级方案。 |
| 先采集后补同意 | 先用最小场景复现，再决定是否升级方案。 |
| 埋点字段无限扩张 | 先用最小场景复现，再决定是否升级方案。 |
| 第三方 SDK 未纳入治理 | 先用最小场景复现，再决定是否升级方案。 |
