## 工程决策

中间人攻击（MITM） 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| 输入处理 | 防护要同时覆盖服务端校验、浏览器策略、输出编码、审计和应急。 | 维护 payload 回归集、安全头检查、依赖审计和权限绕过测试。 |
| 输出编码 | 在“富文本”场景下判断收益、成本、失败路径和回退策略。 | payload 回归集 |
| 权限校验 | 在“后台管理”场景下判断收益、成本、失败路径和回退策略。 | CSP report |
| 浏览器安全头 | 在“第三方脚本接入”场景下判断收益、成本、失败路径和回退策略。 | 依赖审计 |

## 实战路径

推荐按四步推进：

1. **界定场景**：明确当前是新功能接入、旧系统治理、性能优化、安全加固还是线上排障。
2. **建立样例**：用最小代码或配置复现 中间人攻击（MITM） 的核心行为，不直接在完整业务里猜测。
3. **接入约束**：把 输入处理、输出编码、权限校验 写进方案，说明不满足时如何降级。
4. **验证沉淀**：用 安全扫描、payload 回归集、CSP report 形成证据，并把结论回写到团队文档。

type ChapterDecision = {

/**

 * @returns 可写入技术方案或评审记录的决策对象。
 */
function createDecisionRecord(owner: string): ChapterDecision {
  return {
    objective: "中间人攻击（MITM）",
    constraints: ["中间人攻击（MITM）","中间人攻击","MITM","中间人"],

  }
}

## 诊断与验证

| 现象 | 定位动作 | 判断完成的证据 |
|---|---|---|
| 行为与预期不一致 | 回到最小样例，逐步加入真实数据、环境和配置 | 能明确指出差异来自输入、运行时、配置还是依赖 |
| 性能或稳定性退化 | 记录基线，再观察 安全扫描、payload 回归集、CSP report | 指标回到预算内，并且关键路径有回归测试 |
| 团队理解不一致 | 把术语、边界、反例和回退策略写进评审记录 | 新成员可以按同一清单复现和验证 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 追着单个 payload 打补丁，通常会漏掉同类上下文。 | 先用最小场景复现，再决定是否升级方案。 |
| 只修复表面 payload | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| 依赖前端隐藏按钮做权限 | 先用最小场景复现，再决定是否升级方案。 |
| 忽略第三方脚本和依赖链 | 先用最小场景复现，再决定是否升级方案。 |
