## 代码示例

下面的示例用一个轻量检查清单描述如何把「API 设计：REST / GraphQL / gRPC-Web / TRPC」落到工程评审中：

```ts
interface ArticleCheckpoint {
  /** 检查项名称。 */
  title: string
  /** 检查项需要满足的要求。 */
  requirement: string
  /** 当前检查项是否已经完成。 */
  done: boolean
}

/**
 * 筛选仍未完成的检查项。
 * @param checkpoints 当前主题的工程检查清单。
 * @returns 尚未完成的检查项列表。
 */
function getUnfinishedCheckpoints(checkpoints: ArticleCheckpoint[]): ArticleCheckpoint[] {
  return checkpoints.filter((checkpoint) => {
    // 中文注释：只保留未完成项，便于在评审或上线前集中处理风险。
    return !checkpoint.done
  })
}

const checkpoints: ArticleCheckpoint[] = [
  {
    title: "概念边界",
    requirement: "已经说明适用场景、限制条件和替代方案。",
    done: true,
  },
  {
    title: "工程验证",
    requirement: "已经准备测试、日志或性能指标来验证结论。",
    done: false,
  },
]

console.table(getUnfinishedCheckpoints(checkpoints))
```

## 工程实践

API 设计：REST / GraphQL / gRPC-Web / TRPC 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| API 风格 | 先定义对前端稳定的契约，再决定 REST、GraphQL、tRPC、BFF 或 Serverless 边界。 | 用契约测试、超时注入、压测、trace 和结构化日志证明服务可靠。 |
| 中间件 | 在“SSR 数据层”场景下判断收益、成本、失败路径和回退策略。 | trace |
| 数据访问 | 在“实时网关”场景下判断收益、成本、失败路径和回退策略。 | 压测 |
| 鉴权 | 在“文件上传”场景下判断收益、成本、失败路径和回退策略。 | 错误率 |

在真实项目中落地API 设计：REST / GraphQL / gRPC-Web / TRPC时，可以按以下顺序推进：

| 步骤 | 动作 | 验证方式 |
|---|---|---|
| 识别场景 | 明确它解决的是性能、可维护性、兼容性、安全性还是协作问题。 | 写出触发条件和不适用场景。 |
| 建立基线 | 记录当前实现、指标、错误率或维护成本。 | 使用测试、日志、监控或构建报告作为证据。 |
| 小步改造 | 先在低风险路径验证方案，再扩大到核心链路。 | 对比改造前后的行为和指标。 |

| 误区 | 更可靠的处理方式 |
