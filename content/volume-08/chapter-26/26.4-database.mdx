## 章节定位

数据库交互 不再只按“是什么、怎么用”展开，而是放回「第 26 章 Node.js 服务端开发」的知识链路中理解：用 Node 连接浏览器、业务服务和数据层，承担协议适配、聚合编排、鉴权缓存与边缘逻辑。

> **本节要解决的真实问题**：当项目里遇到 数据库交互、数据库、ORM、SQL 相关设计、选型、接入或排障时，能说清楚边界、取舍、验证方式和失败后的回退策略。

| 层次 | 学完后应该能做到 |

| 概念定位 | 解释 数据库交互 在 Node.js 服务端开发 中解决的真实问题 |

| 工程判断 | 为 SQL 选择可验证、可回退的落地方案 |
| 验证闭环 | 围绕“服务边界是什么”建立判断标准 |

## 知识地图

前端全栈不是把后端搬进前端，而是在体验层建立稳定契约、超时边界和可观测服务。

| 层级 | 本节关注点 | 判断证据 |
|---|---|---|
| 问题入口 | 数据库交互、数据库、ORM 会在什么业务场景中暴露 | BFF、SSR 数据层 |

| 工程约束 | API 风格、中间件、数据访问、鉴权 | 兼容性、性能、安全、维护成本和回退方案 |
| 关联知识 | 与同章和跨章内容形成可追踪链路 | 关系链接、示例代码、测试和线上数据 |

## 核心机制

这一节建议沿着“术语 -> 机制 -> 证据”的顺序阅读。先不要急着背 API，先确认每个概念在系统里承担什么职责，以及它失败时会出现什么现象。

| 主题 | 需要理解的机制 | 可观察证据 |
|---|---|---|
| 数据库交互 | Node 层常承担协议适配、聚合编排、鉴权、缓存、实时连接和 SSR 数据准备。 | 用契约测试、超时注入、压测、trace 和结构化日志证明服务可靠。 |
| 数据库 | 围绕“契约如何演进”拆解概念边界、运行机制和失败路径。 | trace |
| ORM | 围绕“失败如何降级”拆解概念边界、运行机制和失败路径。 | 压测 |
| SQL | 围绕“缓存与权限如何协作”拆解概念边界、运行机制和失败路径。 | 错误率 |

<Callout type="tip" title="阅读方法">
读这一节时，建议为每个关键词补一个“最小复现场景”和一个“线上验证信号”。能被复现和验证的知识，才真正能进入工程判断。
</Callout>

## 工程决策

数据库交互 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| API 风格 | 先定义对前端稳定的契约，再决定 REST、GraphQL、tRPC、BFF 或 Serverless 边界。 | 用契约测试、超时注入、压测、trace 和结构化日志证明服务可靠。 |
| 中间件 | 在“SSR 数据层”场景下判断收益、成本、失败路径和回退策略。 | trace |
| 数据访问 | 在“实时网关”场景下判断收益、成本、失败路径和回退策略。 | 压测 |
| 鉴权 | 在“文件上传”场景下判断收益、成本、失败路径和回退策略。 | 错误率 |

## 实战路径

推荐按四步推进：

1. **界定场景**：明确当前是新功能接入、旧系统治理、性能优化、安全加固还是线上排障。
2. **建立样例**：用最小代码或配置复现 数据库交互 的核心行为，不直接在完整业务里猜测。
3. **接入约束**：把 API 风格、中间件、数据访问 写进方案，说明不满足时如何降级。
4. **验证沉淀**：用 结构化日志、trace、压测 形成证据，并把结论回写到团队文档。

```ts
type ChapterDecision = {
  objective: string
  constraints: string[]
  signals: string[]
  rollback: string
}

/**
 * 生成本节主题的落地决策记录。
 * @param owner 负责验证和维护该方案的角色或小组。
 * @returns 可写入技术方案或评审记录的决策对象。
 */
function createDecisionRecord(owner: string): ChapterDecision {
  return {
    objective: "数据库交互",
    constraints: ["数据库交互","数据库","ORM","SQL"],
    signals: ["结构化日志","trace","压测"],
    // 中文注释：回退策略必须在接入前定义，避免上线后只能被动救火。
    rollback: `由 ${owner} 负责准备降级开关、兼容方案和验证清单`,
  }
}

console.table(createDecisionRecord("frontend"))
```

## 诊断与验证

| 现象 | 定位动作 | 判断完成的证据 |
|---|---|---|
| 行为与预期不一致 | 回到最小样例，逐步加入真实数据、环境和配置 | 能明确指出差异来自输入、运行时、配置还是依赖 |
| 性能或稳定性退化 | 记录基线，再观察 结构化日志、trace、压测 | 指标回到预算内，并且关键路径有回归测试 |
| 团队理解不一致 | 把术语、边界、反例和回退策略写进评审记录 | 新成员可以按同一清单复现和验证 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 直接透传后端内部模型，会让前端契约随服务端实现一起震荡。 | 先用最小场景复现，再决定是否升级方案。 |
| 把 BFF 写成新单体 | 先用最小场景复现，再决定是否升级方案。 |
| 忽略超时和限流 | 先用最小场景复现，再决定是否升级方案。 |
| 透传后端内部模型 | 先用最小场景复现，再决定是否升级方案。 |
