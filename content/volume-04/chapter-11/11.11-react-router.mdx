## 代码示例

下面的示例用一个轻量检查清单描述如何把「React Router：v6 新特性、Data API（loader/action/defer）」落到工程评审中：

```ts
interface ArticleCheckpoint {
  /** 检查项名称。 */
  title: string
  /** 检查项需要满足的要求。 */
  requirement: string
  /** 当前检查项是否已经完成。 */
  done: boolean
}

/**
 * 筛选仍未完成的检查项。
 * @param checkpoints 当前主题的工程检查清单。
 * @returns 尚未完成的检查项列表。
 */
function getUnfinishedCheckpoints(checkpoints: ArticleCheckpoint[]): ArticleCheckpoint[] {
  return checkpoints.filter((checkpoint) => {
    // 中文注释：只保留未完成项，便于在评审或上线前集中处理风险。
    return !checkpoint.done
  })
}

const checkpoints: ArticleCheckpoint[] = [
  {
    title: "概念边界",
    requirement: "已经说明适用场景、限制条件和替代方案。",
    done: true,
  },
  {
    title: "工程验证",
    requirement: "已经准备测试、日志或性能指标来验证结论。",
    done: false,
  },
]

console.table(getUnfinishedCheckpoints(checkpoints))
```

## 工程实践

React Router：v6 新特性、Data API（loader/action/defer） 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| 组件边界 | 适合需要深度定制、复杂兼容或 Module Federation 的大型项目；简单应用要评估配置成本。 | 检查 chunk 拆分、sideEffects、缓存命中和 federation 远程模块加载失败路径。 |
| 状态来源 | 先定义对前端稳定的契约，再决定 REST、GraphQL、tRPC、BFF 或 Serverless 边界。 | 用契约测试、超时注入、压测、trace 和结构化日志证明服务可靠。 |
| 渲染时机 | 按状态来源划分本地状态、服务端状态、URL 状态和表单状态。 | 用 Profiler、render 次数、hydration 日志和关键交互耗时验证。 |
| 缓存与失效 | 在“跨页面状态共享”场景下判断收益、成本、失败路径和回退策略。 | bundle 分析 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 只堆 loader/plugin 而不分析模块图，会让构建慢且产物不可解释。 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| 直接透传后端内部模型，会让前端契约随服务端实现一起震荡。 | 先用最小场景复现，再决定是否升级方案。 |
| 把服务端数据、表单状态和 UI 临时状态混进同一个全局 store。 | 先用最小场景复现，再决定是否升级方案。 |
| 把所有数据塞进全局状态 | 先用最小场景复现，再决定是否升级方案。 |
| 在 render 阶段执行副作用 | 先用最小场景复现，再决定是否升级方案。 |
