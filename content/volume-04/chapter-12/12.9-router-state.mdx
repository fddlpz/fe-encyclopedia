## 工程决策

Vue 路由与状态管理：Vue Router 4、Pinia、Vuex 4 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| ref/reactive 选择 | 按状态来源划分本地状态、服务端状态、URL 状态和表单状态。 | 用 Profiler、render 次数、hydration 日志和关键交互耗时验证。 |
| computed/watch 边界 | 派生状态优先 computed，副作用用 watch，跨页面状态再引入 store。 | 用 Vue DevTools 检查依赖、更新次数、组件边界和 hydration mismatch。 |
| Composables 抽象 | 在“组件库封装”场景下判断收益、成本、失败路径和回退策略。 | watch 触发次数 |
| Pinia 与路由状态 | 在“Nuxt 页面渲染”场景下判断收益、成本、失败路径和回退策略。 | bundle 分析 |

## 实战路径

推荐按四步推进：

1. **界定场景**：明确当前是新功能接入、旧系统治理、性能优化、安全加固还是线上排障。
2. **建立样例**：用最小代码或配置复现 Vue 路由与状态管理 的核心行为，不直接在完整业务里猜测。
3. **接入约束**：把 ref/reactive 选择、computed/watch 边界、Composables 抽象 写进方案，说明不满足时如何降级。
4. **验证沉淀**：用 Vue DevTools、组件更新高亮、watch 触发次数 形成证据，并把结论回写到团队文档。

## 诊断与验证

| 现象 | 定位动作 | 判断完成的证据 |
|---|---|---|
| 行为与预期不一致 | 回到最小样例，逐步加入真实数据、环境和配置 | 能明确指出差异来自输入、运行时、配置还是依赖 |
| 性能或稳定性退化 | 记录基线，再观察 Vue DevTools、组件更新高亮、watch 触发次数 | 指标回到预算内，并且关键路径有回归测试 |
| 团队理解不一致 | 把术语、边界、反例和回退策略写进评审记录 | 新成员可以按同一清单复现和验证 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 把服务端数据、表单状态和 UI 临时状态混进同一个全局 store。 | 先用最小场景复现，再决定是否升级方案。 |
| 深度 watch 大对象会让性能和可预测性同时变差。 | 先用最小场景复现，再决定是否升级方案。 |
| 解构 reactive 后丢失响应性 | 先用最小场景复现，再决定是否升级方案。 |
| 滥用深度 watch | 先用最小场景复现，再决定是否升级方案。 |
