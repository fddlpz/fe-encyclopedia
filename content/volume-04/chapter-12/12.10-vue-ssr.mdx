下面的示例用一个轻量检查清单描述如何把「Vue SSR：Nuxt 3、Vite-SSG」落到工程评审中：

```ts
interface ArticleCheckpoint {
  /** 检查项名称。 */
  title: string
  /** 检查项需要满足的要求。 */
  requirement: string
  /** 当前检查项是否已经完成。 */
  done: boolean
}

/**
 * 筛选仍未完成的检查项。
 * @param checkpoints 当前主题的工程检查清单。
 * @returns 尚未完成的检查项列表。
 */
function getUnfinishedCheckpoints(checkpoints: ArticleCheckpoint[]): ArticleCheckpoint[] {
  return checkpoints.filter((checkpoint) => {
    // 中文注释：只保留未完成项，便于在评审或上线前集中处理风险。
    return !checkpoint.done
  })
}

const checkpoints: ArticleCheckpoint[] = [
  {
    title: "概念边界",
    requirement: "已经说明适用场景、限制条件和替代方案。",
    done: true,
  },
  {
    title: "工程验证",
    requirement: "已经准备测试、日志或性能指标来验证结论。",
    done: false,
  },
]

console.table(getUnfinishedCheckpoints(checkpoints))
```

## 工程实践

Vue SSR：Nuxt 3、Vite-SSG 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| ref/reactive 选择 | 选型时同时比较冷启动、HMR、生产产物、插件生态和团队迁移成本。 | 用同一业务切片比较 CI 构建时间、产物体积、source map 和兼容目标。 |
| computed/watch 边界 | 派生状态优先 computed，副作用用 watch，跨页面状态再引入 store。 | 用 Vue DevTools 检查依赖、更新次数、组件边界和 hydration mismatch。 |
| Composables 抽象 | 先定义问题要回答什么，再决定采样、字段、告警和隐私边界。 | 看分位数、版本维度、错误分组、trace 串联和修复后回归。 |
| Pinia 与路由状态 | 在“Nuxt 页面渲染”场景下判断收益、成本、失败路径和回退策略。 | bundle 分析 |

| 误区 | 更可靠的处理方式 |
