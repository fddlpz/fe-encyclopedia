## 章节定位

路由与表单：Angular Router、Reactive Forms vs Template-driven Forms 不再只按“是什么、怎么用”展开，而是放回「第 13 章 Angular 原理与生态」的知识链路中理解：把依赖注入、模板编译、变更检测、路由表单和 RxJS 看成一套企业应用约束系统。

> **本节要解决的真实问题**：当项目里遇到 路由与表单、Angular、Router、Reactive 相关设计、选型、接入或排障时，能说清楚边界、取舍、验证方式和失败后的回退策略。

## 核心机制

这一节建议沿着“术语 -> 机制 -> 证据”的顺序阅读。先不要急着背 API，先确认每个概念在系统里承担什么职责，以及它失败时会出现什么现象。

| 主题 | 需要理解的机制 | 可观察证据 |
|---|---|---|
| 路由与表单 | Node 层常承担协议适配、聚合编排、鉴权、缓存、实时连接和 SSR 数据准备。 | 用契约测试、超时注入、压测、trace 和结构化日志证明服务可靠。 |
| Angular | React 把状态快照映射为组件树，再通过 Fiber 调度、提交阶段和生态库管理复杂 UI。 | 用 Profiler、render 次数、hydration 日志和关键交互耗时验证。 |
| Router | Vue 通过响应式依赖收集、模板编译和更新队列把状态变化同步到视图。 | 用 Vue DevTools 检查依赖、更新次数、组件边界和 hydration mismatch。 |
| Reactive | 围绕“表单和路由状态如何建模”拆解概念边界、运行机制和失败路径。 | bundle 分析 |

## 工程决策

路由与表单：Angular Router、Reactive Forms vs Template-driven Forms 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| Standalone/NgModule | 先定义对前端稳定的契约，再决定 REST、GraphQL、tRPC、BFF 或 Serverless 边界。 | 用契约测试、超时注入、压测、trace 和结构化日志证明服务可靠。 |
| Default/OnPush | 按状态来源划分本地状态、服务端状态、URL 状态和表单状态。 | 用 Profiler、render 次数、hydration 日志和关键交互耗时验证。 |
| Reactive Forms | 派生状态优先 computed，副作用用 watch，跨页面状态再引入 store。 | 用 Vue DevTools 检查依赖、更新次数、组件边界和 hydration mismatch。 |
| RxJS 操作符 | 在“多团队企业应用”场景下判断收益、成本、失败路径和回退策略。 | bundle 分析 |

## 实战路径

推荐按四步推进：

1. **界定场景**：明确当前是新功能接入、旧系统治理、性能优化、安全加固还是线上排障。
2. **建立样例**：用最小代码或配置复现 路由与表单 的核心行为，不直接在完整业务里猜测。
3. **接入约束**：把 Standalone/NgModule、Default/OnPush、Reactive Forms 写进方案，说明不满足时如何降级。
4. **验证沉淀**：用 Angular DevTools、change detection cycles、subscription 泄漏 形成证据，并把结论回写到团队文档。

## 诊断与验证

| 现象 | 定位动作 | 判断完成的证据 |
|---|---|---|
| 行为与预期不一致 | 回到最小样例，逐步加入真实数据、环境和配置 | 能明确指出差异来自输入、运行时、配置还是依赖 |
| 性能或稳定性退化 | 记录基线，再观察 Angular DevTools、change detection cycles、subscription 泄漏 | 指标回到预算内，并且关键路径有回归测试 |
| 团队理解不一致 | 把术语、边界、反例和回退策略写进评审记录 | 新成员可以按同一清单复现和验证 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 直接透传后端内部模型，会让前端契约随服务端实现一起震荡。 | 先用最小场景复现，再决定是否升级方案。 |
| 把服务端数据、表单状态和 UI 临时状态混进同一个全局 store。 | 先用最小场景复现，再决定是否升级方案。 |
| 深度 watch 大对象会让性能和可预测性同时变差。 | 先用最小场景复现，再决定是否升级方案。 |
| 不了解 Zone.js 与变更检测关系 | 先用最小场景复现，再决定是否升级方案。 |
| 模板里放复杂计算 | 先用最小场景复现，再决定是否升级方案。 |
