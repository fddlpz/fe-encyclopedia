## 工程决策

Qwik：可恢复性（Resumability） 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| 首屏加载 | 用真实业务切片比较首屏、交互恢复、数据获取、部署和生态缺口。 | 同时看 LCP、INP、bundle、hydration 或 resumability 成本。 |
| 交互恢复 | 在“营销页”场景下判断收益、成本、失败路径和回退策略。 | LCP/INP/CLS |
| 数据获取 | 在“高互动应用试点”场景下判断收益、成本、失败路径和回退策略。 | bundle 与 hydration 成本 |
| 部署环境 | 在“局部性能瓶颈改造”场景下判断收益、成本、失败路径和回退策略。 | 开发效率反馈 |

## 实战路径

推荐按四步推进：

1. **界定场景**：明确当前是新功能接入、旧系统治理、性能优化、安全加固还是线上排障。
2. **建立样例**：用最小代码或配置复现 Qwik 的核心行为，不直接在完整业务里猜测。
3. **接入约束**：把 首屏加载、交互恢复、数据获取 写进方案，说明不满足时如何降级。
4. **验证沉淀**：用 真实业务切片 benchmark、LCP/INP/CLS、bundle 与 hydration 成本 形成证据，并把结论回写到团队文档。

```ts
type ChapterDecision = {
  objective: string
  constraints: string[]
  signals: string[]
  rollback: string
}

/**
 * 生成本节主题的落地决策记录。
 * @param owner 负责验证和维护该方案的角色或小组。
 * @returns 可写入技术方案或评审记录的决策对象。
 */
function createDecisionRecord(owner: string): ChapterDecision {
  return {
    objective: "可恢复性（Resumability）",
    constraints: ["Qwik","可恢复性","Resumability","性能"],
    signals: ["真实业务切片 benchmark","LCP/INP/CLS","bundle 与 hydration 成本"],
    // 中文注释：回退策略必须在接入前定义，避免上线后只能被动救火。
    rollback: `由 ${owner} 负责准备降级开关、兼容方案和验证清单`,
  }
}

console.table(createDecisionRecord("frontend"))
```

## 诊断与验证

| 现象 | 定位动作 | 判断完成的证据 |
|---|---|---|
| 行为与预期不一致 | 回到最小样例，逐步加入真实数据、环境和配置 | 能明确指出差异来自输入、运行时、配置还是依赖 |
| 性能或稳定性退化 | 记录基线，再观察 真实业务切片 benchmark、LCP/INP/CLS、bundle 与 hydration 成本 | 指标回到预算内，并且关键路径有回归测试 |
| 团队理解不一致 | 把术语、边界、反例和回退策略写进评审记录 | 新成员可以按同一清单复现和验证 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 迁移前没有试点和回退方案，会把框架风险变成产品风险。 | 先用最小场景复现，再决定是否升级方案。 |
| 把宣传语等同于收益 | 先用最小场景复现，再决定是否升级方案。 |
| 只看单项 benchmark | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| 忽略生态和长期维护成本 | 先用最小场景复现，再决定是否升级方案。 |
