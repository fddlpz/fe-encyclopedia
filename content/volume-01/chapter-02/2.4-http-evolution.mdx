## 背景与问题

HTTP 的演进主线是：语义基本保持稳定，传输效率、连接复用、头部压缩和多路复用能力不断增强。`GET`、`POST`、状态码、字段语义属于 HTTP 语义层；HTTP/1.1、HTTP/2、HTTP/3 是这些语义在网络上的不同表达方式。

| 版本 | 核心变化 | 前端影响 |
|---|---|---|
| HTTP/0.9 | 只有非常简单的 `GET` 和 HTML 响应 | 只适合理解历史 |
| HTTP/1.0 | 引入请求头、响应头、状态码、内容类型 | 奠定现代 Web 基础 |
| HTTP/1.1 | 默认长连接、Host、分块传输、缓存语义增强 | 多域名拆分、连接复用、缓存控制 |
| HTTP/2 | 二进制帧、多路复用、HPACK 头部压缩、流优先级 | 减少域名分片，关注资源优先级 |
| HTTP/3 | HTTP 语义映射到 QUIC，基于 UDP，多路复用不再受 TCP 丢包队头阻塞影响 | 弱网和移动网络可能收益明显，但要验证 CDN 和回退 |

在前端工程中，HTTP 协议演进：HTTP/0.9 → HTTP/1.0 → HTTP/1.1 → HTTP/2 → HTTP/3通常不是单独存在的知识点，而是连接语言特性、浏览器能力、框架抽象和工程约束的判断入口。理解它的背景，可以帮助我们回答三个问题：为什么需要这个机制、它解决了什么边界问题，以及在什么场景下不应该过度使用。

## 工作原理

HTTP/2 和 HTTP/3 并没有改变 `GET /api/users` 的业务语义，它们改变的是字节如何在连接上组织和传输。HTTP/2 在一个 TCP 连接上承载多个并发流；如果底层 TCP 丢包，同连接上的流仍会等待 TCP 重传。HTTP/3 使用 QUIC，在传输层提供独立流，单个流丢包通常不会阻塞其他流的数据交付。

## 代码示例

前端日常最常处理的是 HTTP 方法、状态码、缓存头和资源加载策略。

```http
GET /assets/app.8f3a1.js HTTP/2
Host: static.example.com
Accept-Encoding: br, gzip
Cache-Control: max-age=31536000, immutable
```

常见方法和语义：

| 方法 | 典型用途 | 注意点 |
|---|---|---|
| `GET` | 读取资源 | 应尽量无副作用，可被缓存 |
| `POST` | 创建资源或提交动作 | 不默认幂等，避免被错误重试 |
| `PUT` | 整体替换资源 | 通常要求幂等 |
| `PATCH` | 局部更新资源 | 需要明确补丁格式 |
| `DELETE` | 删除资源 | 权限和幂等语义要明确 |
| `OPTIONS` | 预检或能力查询 | CORS 预检会使用 |

状态码要按类别理解：

| 类别 | 含义 | 示例 |
|---|---|---|
| `2xx` | 成功 | `200 OK`、`204 No Content` |
| `3xx` | 重定向或缓存协商 | `301`、`302`、`304` |
| `4xx` | 客户端请求问题 | `400`、`401`、`403`、`404`、`409`、`429` |
| `5xx` | 服务端处理问题 | `500`、`502`、`503`、`504` |

用浏览器或命令行可以检查真实协议版本：

```bash
# -I 只请求响应头；--http3 需要当前 curl 构建支持 HTTP/3
curl -I --http2 https://example.com/
curl -I --http3 https://example.com/
```

## 工程实践

静态资源站点的协议和缓存策略通常可以这样设计：

1. HTML 使用短缓存或协商缓存，保证入口文件能及时更新。
2. 带 hash 的 JS/CSS/图片使用长期强缓存。
3. 同源资源优先复用 HTTP/2 或 HTTP/3 连接，减少不必要的域名分片。
4. API 返回明确的状态码和错误结构，避免所有错误都包装成 `200`。
5. 对跨域接口明确 CORS 头，减少无意义的预检。

```http
HTTP/2 200
content-type: application/javascript; charset=utf-8
cache-control: public, max-age=31536000, immutable
content-encoding: br
```

一个可靠的 API 错误响应应同时给人和程序读取：

```json
{
  "code": "RATE_LIMITED",
  "message": "请求过于频繁，请稍后重试",
  "requestId": "req_01HZ..."
}
```

前端请求层可以把协议错误、业务错误和网络失败分开处理：

```ts
type ApiError = {
  code: string
  message: string
  requestId?: string
}

async function requestJson<T>(url: string): Promise<T> {
  const response = await fetch(url)

  if (!response.ok) {
    // 中文注释：HTTP 非 2xx 先按协议错误处理，再解析业务错误体。
    const error = (await response.json().catch(() => null)) as ApiError | null
    throw new Error(error?.message ?? `HTTP ${response.status}`)
  }

  return response.json() as Promise<T>
}
```

## 常见误区

| 常见误区 | 风险 | 建议 |
|---|---|---|
| HTTP/2 下继续大量域名分片 | 连接复用收益下降，TLS 和连接成本增加 | 优先减少不必要的域名 |
| 把所有业务错误都返回 `200` | 监控、缓存、重试和网关策略都会失真 | 使用合适状态码和稳定错误结构 |
| 认为 HTTP/3 永远更快 | UDP 被阻断、网关不支持、回退策略不佳时可能没有收益 | 通过真实用户监控对比 |
| 忽略缓存语义 | 入口文件长缓存会导致用户拿到旧版本 | 区分 HTML 和带 hash 静态资源 |
| 滥用 `POST` | 无法利用缓存和幂等重试语义 | 按资源语义选择方法 |
