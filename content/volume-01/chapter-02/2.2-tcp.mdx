## 背景与问题

TCP 是面向连接的可靠字节流协议。它不理解 HTTP、JSON 或图片，只负责在两个端点之间提供“有序、可靠、无重复”的字节传输。可靠性来自序列号、确认应答、重传、滑动窗口、流量控制和拥塞控制。

三次握手的目标不是“发送三次消息”这么简单，而是让双方确认四件事：客户端能发、服务端能收、服务端能发、客户端能收，并协商初始序列号。

```text
客户端                                      服务端
SYN, seq=x ------------------------------>
                         <-------------- SYN+ACK, seq=y, ack=x+1
ACK, ack=y+1 ---------------------------->
连接建立，应用层开始发送数据
```

四次挥手的核心是 TCP 的全双工特性：一端关闭发送方向，并不代表另一端不能继续发送剩余数据。

## 工作原理

```text

SYN, seq=x ------------------------------>
                         <-------------- SYN+ACK, seq=y, ack=x+1
ACK, ack=y+1 ---------------------------->
连接建立，应用层开始发送数据
```

四次挥手的核心是 TCP 的全双工特性：一端关闭发送方向，并不代表另一端不能继续发送剩余数据。

```text
主动关闭方                                  被动关闭方
FIN -------------------------------------->
                         <-------------- ACK
                         <-------------- FIN
ACK -------------------------------------->
进入 TIME_WAIT，等待足够时间后释放连接
```

`TIME_WAIT` 通常出现在主动关闭连接的一方。它的作用是确保最后一个 ACK 有机会被对端收到，并避免旧连接的延迟报文污染后续复用的四元组。

## 代码示例

在前端排查里，TCP 常通过间接信号出现。

| 机制 | 含义 | 前端可观察现象 |
|---|---|---|
| 三次握手 | 建立连接并同步初始序列号 | DevTools 中 `Initial connection` 耗时 |
| 滑动窗口 | 接收方声明还能接收多少数据 | 大文件下载受接收端处理能力影响 |
| 拥塞控制 | 根据网络拥塞程度调整发送速率 | 弱网或跨地域首包后吞吐逐步上升 |
| 重传 | 报文丢失后重新发送 | 请求偶发变慢、RTT 抖动、抓包出现重传 |
| Nagle 算法 | 合并小包以减少报文数量 | 小消息低延迟场景可能需要服务端关闭 |
| Keep-Alive | 复用已有连接 | 后续请求省掉 TCP/TLS 建连成本 |

用 `curl` 或浏览器瀑布图可以拆出 TCP 建连耗时：

```bash
# connect 表示 TCP 连接建立完成的时间点
curl -o NUL -s -w "connect=%{time_connect}\ntls=%{time_appconnect}\ntotal=%{time_total}\n" https://example.com/
```

如果 `time_connect` 明显高，优先检查访问距离、网络质量、代理、负载均衡和是否频繁新建连接；如果只有首个请求高、后续请求低，通常说明连接复用正在生效。

## 工程实践

前端项目里最常见的 TCP 优化不是调 TCP 参数，而是减少不必要的新连接。

1. 合理合并关键域名，避免首屏资源分散到过多主机。
2. 使用 HTTP Keep-Alive，让同源请求复用连接。
3. 对关键跨域域名使用 `preconnect`，提前完成 DNS、TCP 和 TLS。
4. 避免大量小请求阻塞关键路径，优先加载首屏必要资源。
5. 对实时通信服务端配置心跳和空闲超时，避免代理静默断开连接。

```html
<!-- 仅对首屏确定需要的关键源使用 preconnect -->
<link rel="preconnect" href="https://api.example.com" crossorigin />
```

服务端在低延迟小消息场景中可能会显式关闭 Nagle 算法，例如 Node.js 的 TCP socket 提供 `setNoDelay(true)`。前端工程师不一定直接写这段代码，但需要知道为什么 WebSocket 聊天、行情推送等场景会关注小包延迟。

```ts
import net from "node:net"

const server = net.createServer((socket) => {
  // 中文注释：低延迟小消息场景中关闭 Nagle，减少等待合包带来的延迟。
  socket.setNoDelay(true)
})

server.listen(9000)
```

## 常见误区

| 常见误区 | 风险 | 建议 |
|---|---|---|
| 把 TCP 连接数加大当作万能优化 | 连接越多，握手、拥塞控制和服务端资源成本越高 | 先确认瓶颈，再决定拆域名或合并资源 |
| 忽略 TIME_WAIT | 高并发短连接会堆积大量关闭中的连接 | 使用连接复用、连接池和合理的服务端关闭策略 |
| 认为 HTTP/2 消除了所有队头阻塞 | HTTP/2 消除了应用层请求队头阻塞，但底层 TCP 丢包仍会影响同连接所有流 | 丢包明显时评估 HTTP/3/QUIC |
| 用平均耗时判断网络质量 | TCP 重传和拥塞往往体现在长尾请求 | 同时看 P95/P99、失败率和地区维度 |
| 只在本地验证 | 本地网络不会暴露跨运营商、跨地域、移动网络问题 | 用真实用户监控和多地区探针验证 |
