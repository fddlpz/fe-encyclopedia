## 原理

WebSocket 在一次 HTTP/1.1 Upgrade 握手之后，把连接升级为全双工消息通道。它适合浏览器和服务端之间持续双向通信，例如聊天、协同编辑、行情推送、在线状态同步。

握手请求包含几个关键头：

```http
GET /socket HTTP/1.1
Host: example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
Origin: https://app.example.com
```

服务端确认升级：

```http
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

升级完成后，通信不再是普通 HTTP 请求响应，而是 WebSocket 帧。帧可以承载文本、二进制、关闭帧、Ping、Pong 等。浏览器端脚本不能直接发送协议级 Ping 帧，因此业务上常用应用层心跳消息。

<Callout type="info" title="WebSocket 和 HTTP 的边界">
WebSocket 借用 HTTP 完成握手，但升级后不是“长轮询 HTTP”。它是一条持续存在的双向连接，必须单独设计鉴权、心跳、重连和限流。
</Callout>

## 用法

浏览器端最小 WebSocket 客户端如下：

```ts
const socket = new WebSocket("wss://example.com/socket")

socket.addEventListener("open", () => {
  // 中文注释：连接建立后再发送订阅消息，避免在 CONNECTING 状态发送失败。
  socket.send(JSON.stringify({ type: "subscribe", channel: "notice" }))
})

socket.addEventListener("message", (event) => {
  const message = JSON.parse(event.data)
  console.log("message", message)
})

socket.addEventListener("close", (event) => {
  // 中文注释：根据关闭码和业务状态决定是否重连。
  console.log("closed", event.code, event.reason)
})
```

生产项目通常需要封装重连和心跳：

```ts
type SocketMessage = {
  type: string
  payload?: unknown
}

function createSocket(url: string) {
  let socket: WebSocket | null = null
  let retryCount = 0
  let heartbeatTimer: ReturnType<typeof setInterval> | null = null

  const connect = () => {
    socket = new WebSocket(url)

    socket.addEventListener("open", () => {
      retryCount = 0
      heartbeatTimer = setInterval(() => {
        // 中文注释：浏览器不能发送协议级 Ping，这里发送应用层心跳。
        socket?.send(JSON.stringify({ type: "ping" }))
      }, 30000)
    })

    socket.addEventListener("close", () => {
      if (heartbeatTimer) clearInterval(heartbeatTimer)
      const delay = Math.min(1000 * 2 ** retryCount, 30000)
      retryCount += 1
      window.setTimeout(connect, delay)
    })
  }

  connect()

  return {
    send(message: SocketMessage) {
      if (socket?.readyState === WebSocket.OPEN) {
        socket.send(JSON.stringify(message))
      }
    },
  }
}
```

## 实践

WebSocket、SSE 和轮询应按交互模型选择。

| 方案 | 通信方向 | 优点 | 代价 |
|---|---|---|---|
| WebSocket | 双向 | 低延迟、适合协作和实时交互 | 连接治理、扩容、心跳复杂 |
| SSE | 服务端到客户端 | 简单、自动重连、走 HTTP | 只能单向推送文本事件 |
| 短轮询 | 客户端定时请求 | 实现简单、兼容性强 | 延迟和请求成本较高 |
| 长轮询 | 服务端挂起请求 | 兼容旧环境 | 服务端连接占用和实现复杂 |

WebSocket 上线要重点处理：

1. **鉴权**：不要只依赖 URL 参数明文 token，优先使用短期票据或握手阶段校验。
2. **Origin 校验**：服务端不能默认信任所有来源。
3. **心跳**：客户端和服务端都要能发现半开连接。
4. **重连退避**：避免服务端故障时所有客户端同时重连。
5. **消息幂等**：重连后可能重复订阅或重复收到消息。
6. **背压**：客户端处理不过来时需要丢弃、合并或降级消息。

## 陷阱

| 常见误区 | 风险 | 建议 |
|---|---|---|
| 没有心跳 | NAT、代理或负载均衡可能静默断开连接 | 设计应用层心跳和超时 |
| 固定间隔无限重连 | 故障时形成重连风暴 | 指数退避并加随机抖动 |
| 不校验 Origin | 可能被非预期站点发起连接 | 服务端校验 Origin 和鉴权 |
| 把 WebSocket 当消息队列 | 客户端连接不可靠，无法天然保证消息持久化 | 需要 ACK、补偿和离线拉取 |
| 页面隐藏仍高频推送 | 移动端耗电和后台限制 | 页面隐藏时降低频率或暂停非关键订阅 |
