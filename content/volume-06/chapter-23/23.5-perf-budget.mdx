## 核心概念

可观测性不是埋点越多越好，而是用最小必要数据回答线上发生了什么、影响谁、为什么发生。

| 层级 | 本节关注点 | 判断证据 |
|---|---|---|
| 问题入口 | 性能预算、预算、体积控制 会在什么业务场景中暴露 | 新功能上线、性能劣化定位 |
| 运行机制 | 指标定义是否稳定；采样是否合理；隐私边界是否清楚 | RUM 分位数、错误分组 |
| 工程约束 | 采集粒度、采样率、Source Map、Trace Context | 兼容性、性能、安全、维护成本和回退方案 |
| 关联知识 | 与同章和跨章内容形成可追踪链路 | 关系链接、示例代码、测试和线上数据 |

## 代码示例

下面的示例用一个轻量检查清单描述如何把「性能预算」落到工程评审中：

```ts
interface ArticleCheckpoint {
  /** 检查项名称。 */
  title: string
  /** 检查项需要满足的要求。 */
  requirement: string
  /** 当前检查项是否已经完成。 */
  done: boolean
}

/**
 * 筛选仍未完成的检查项。
 * @param checkpoints 当前主题的工程检查清单。
 * @returns 尚未完成的检查项列表。
 */
function getUnfinishedCheckpoints(checkpoints: ArticleCheckpoint[]): ArticleCheckpoint[] {
  return checkpoints.filter((checkpoint) => {
    // 中文注释：只保留未完成项，便于在评审或上线前集中处理风险。
    return !checkpoint.done
  })
}

const checkpoints: ArticleCheckpoint[] = [
  {
    title: "概念边界",
    requirement: "已经说明适用场景、限制条件和替代方案。",
    done: true,
  },
  {
    title: "工程验证",
    requirement: "已经准备测试、日志或性能指标来验证结论。",
    done: false,
  },
]

console.table(getUnfinishedCheckpoints(checkpoints))
```

## 工程实践

性能预算 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| 采集粒度 | 先定义问题要回答什么，再决定采样、字段、告警和隐私边界。 | 看分位数、版本维度、错误分组、trace 串联和修复后回归。 |
| 采样率 | 在“性能劣化定位”场景下判断收益、成本、失败路径和回退策略。 | 错误分组 |
| Source Map | 在“错误爆发”场景下判断收益、成本、失败路径和回退策略。 | trace id |
| Trace Context | 在“用户路径分析”场景下判断收益、成本、失败路径和回退策略。 | 日志上下文 |

在真实项目中落地性能预算时，可以按以下顺序推进：

| 步骤 | 动作 | 验证方式 |
|---|---|---|
| 识别场景 | 明确它解决的是性能、可维护性、兼容性、安全性还是协作问题。 | 写出触发条件和不适用场景。 |
| 建立基线 | 记录当前实现、指标、错误率或维护成本。 | 使用测试、日志、监控或构建报告作为证据。 |
| 小步改造 | 先在低风险路径验证方案，再扩大到核心链路。 | 对比改造前后的行为和指标。 |
| 固化规范 | 将经验沉淀为 lint、测试、文档或评审清单。 | 新增场景能复用同一套判断标准。 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 采集很多但无法定位责任边界，等于把噪音写进系统。 | 先用最小场景复现，再决定是否升级方案。 |
| 无节制采集用户数据 | 先用最小场景复现，再决定是否升级方案。 |
| 只收集错误没有版本和路径 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| 告警噪音过大 | 先用最小场景复现，再决定是否升级方案。 |
