## 章节定位

错误监控：Sentry / Source Map 不再只按“是什么、怎么用”展开，而是放回「第 23 章 监控与可观测性」的知识链路中理解：把真实用户性能、错误、日志、链路追踪和体验分析连接成发现、定位、修复、验证闭环。

> **本节要解决的真实问题**：当项目里遇到 错误监控、Sentry、Source、Map 相关设计、选型、接入或排障时，能说清楚边界、取舍、验证方式和失败后的回退策略。

| 层次 | 学完后应该能做到 |

| 工程判断 | 为 Map 选择可验证、可回退的落地方案 |
| 验证闭环 | 围绕“指标定义是否稳定”建立判断标准 |

## 知识地图

可观测性不是埋点越多越好，而是用最小必要数据回答线上发生了什么、影响谁、为什么发生。

| 层级 | 本节关注点 | 判断证据 |
|---|---|---|
| 问题入口 | 错误监控、Sentry、Source 会在什么业务场景中暴露 | 新功能上线、性能劣化定位 |
| 运行机制 | 指标定义是否稳定；采样是否合理；隐私边界是否清楚 | RUM 分位数、错误分组 |
| 工程约束 | 采集粒度、采样率、Source Map、Trace Context | 兼容性、性能、安全、维护成本和回退方案 |
| 关联知识 | 与同章和跨章内容形成可追踪链路 | 关系链接、示例代码、测试和线上数据 |

## 核心机制

这一节建议沿着“术语 -> 机制 -> 证据”的顺序阅读。先不要急着背 API，先确认每个概念在系统里承担什么职责，以及它失败时会出现什么现象。

| 主题 | 需要理解的机制 | 可观察证据 |
|---|---|---|
| 错误监控 | 可观测性用指标、事件、日志和链路把线上体验变成可解释证据。 | 看分位数、版本维度、错误分组、trace 串联和修复后回归。 |
| Sentry | 围绕“采样是否合理”拆解概念边界、运行机制和失败路径。 | 错误分组 |
| Source | 围绕“隐私边界是否清楚”拆解概念边界、运行机制和失败路径。 | trace id |
| Map | 围绕“告警是否可行动”拆解概念边界、运行机制和失败路径。 | 日志上下文 |

</Callout>

## 工程决策

错误监控：Sentry / Source Map 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| 采集粒度 | 先定义问题要回答什么，再决定采样、字段、告警和隐私边界。 | 看分位数、版本维度、错误分组、trace 串联和修复后回归。 |
| 采样率 | 在“性能劣化定位”场景下判断收益、成本、失败路径和回退策略。 | 错误分组 |
| Source Map | 在“错误爆发”场景下判断收益、成本、失败路径和回退策略。 | trace id |
| Trace Context | 在“用户路径分析”场景下判断收益、成本、失败路径和回退策略。 | 日志上下文 |

## 实战路径

推荐按四步推进：

1. **界定场景**：明确当前是新功能接入、旧系统治理、性能优化、安全加固还是线上排障。
2. **建立样例**：用最小代码或配置复现 错误监控 的核心行为，不直接在完整业务里猜测。
3. **接入约束**：把 采集粒度、采样率、Source Map 写进方案，说明不满足时如何降级。
4. **验证沉淀**：用 RUM 分位数、错误分组、trace id 形成证据，并把结论回写到团队文档。

```ts
type ChapterDecision = {
  objective: string
  constraints: string[]
  signals: string[]
  rollback: string
}

/**
 * 生成本节主题的落地决策记录。
 * @param owner 负责验证和维护该方案的角色或小组。
 * @returns 可写入技术方案或评审记录的决策对象。
 */
function createDecisionRecord(owner: string): ChapterDecision {
  return {
    objective: "Sentry / Source Map",
    constraints: ["错误监控","Sentry","Source","Map"],
    signals: ["RUM 分位数","错误分组","trace id"],
    // 中文注释：回退策略必须在接入前定义，避免上线后只能被动救火。
    rollback: `由 ${owner} 负责准备降级开关、兼容方案和验证清单`,
  }
}

console.table(createDecisionRecord("frontend"))
```

## 诊断与验证

| 现象 | 定位动作 | 判断完成的证据 |
|---|---|---|
| 行为与预期不一致 | 回到最小样例，逐步加入真实数据、环境和配置 | 能明确指出差异来自输入、运行时、配置还是依赖 |
| 性能或稳定性退化 | 记录基线，再观察 RUM 分位数、错误分组、trace id | 指标回到预算内，并且关键路径有回归测试 |
| 团队理解不一致 | 把术语、边界、反例和回退策略写进评审记录 | 新成员可以按同一清单复现和验证 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 采集很多但无法定位责任边界，等于把噪音写进系统。 | 先用最小场景复现，再决定是否升级方案。 |
| 无节制采集用户数据 | 先用最小场景复现，再决定是否升级方案。 |
| 只收集错误没有版本和路径 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| 告警噪音过大 | 先用最小场景复现，再决定是否升级方案。 |
