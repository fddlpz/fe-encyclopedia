## 背景与问题

TypeScript 的对象类型是结构化类型：只要形状兼容，就可以赋值，不要求显式继承关系。`interface` 和 `type` 都能描述对象结构，但能力边界不同。

在前端工程中，基本类型与复合类型：interface vs type、索引签名、联合类型、交叉类型、元组通常不是单独存在的知识点，而是连接语言特性、浏览器能力、框架抽象和工程约束的判断入口。理解它的背景，可以帮助我们回答三个问题：为什么需要这个机制、它解决了什么边界问题，以及在什么场景下不应该过度使用。

## 工作原理

| 主题 | 关键点 | 建议 |
|---|---|---|
| `interface` | 可声明合并，可 `extends` | 公共对象模型、类契约、库类型扩展 |
| `type` | 可表达联合、交叉、条件、元组和映射结果 | 组合类型、函数类型、字面量联合 |
| 索引签名 | 描述动态 key | 配合 `noUncheckedIndexedAccess` 使用 |
| 联合类型 | 多种可能之一 | 用判别字段建模状态 |
| 交叉类型 | 多种结构叠加 | 用于组合能力，避免冲突字段 |
| 元组 | 固定长度和位置语义的数组 | 适合坐标、返回值、命令参数 |

核心取舍：`interface` 更适合可扩展对象契约，`type` 更适合类型表达式和组合。团队不必强行统一所有场景，但需要形成清晰规则。

## 代码示例

对象模型优先 `interface`：

```ts
interface Article {
  id: string
  title: string
  tags: string[]
}

interface FeaturedArticle extends Article {
  featuredAt: string
}
```

状态模型优先判别联合：

```ts
type RequestState<T> =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; error: Error }

function getMessage(state: RequestState<string[]>) {
  if (state.status === "success") {
    // 中文注释：判别字段收窄后，data 只在 success 分支可见。
    return `共 ${state.data.length} 条`
  }

  return state.status
}
```

动态字典要让值类型包含缺失可能：

```ts
type FeatureFlags = Record<string, boolean | undefined>

const flags: FeatureFlags = {}

if (flags.newEditor === true) {
  // 中文注释：显式判断 true，避免 undefined 被当作 false 误判业务状态。
  console.log("启用新编辑器")
}
```

元组要保留位置语义：

```ts
type Point = readonly [x: number, y: number]

function distance([x, y]: Point) {
  return Math.sqrt(x * x + y * y)
}
```

## 工程实践

类型建模时，先列出业务状态，再决定类型结构。对于表单、请求、权限和流程状态，判别联合通常比多个布尔字段更安全。

```ts
type Payment =
  | { kind: "card"; last4: string }
  | { kind: "wallet"; provider: "alipay" | "wechat" }
  | { kind: "invoice"; taxId: string }

function paymentLabel(payment: Payment) {
  switch (payment.kind) {
    case "card":
      return `银行卡 ${payment.last4}`
    case "wallet":
      return payment.provider
    case "invoice":
      return payment.taxId
  }
}
```

API 响应类型不要直接复用 UI 状态类型。接口字段反映后端契约，UI 类型反映页面状态，两者中间可以用 mapper 转换。

## 常见误区

| 常见问题 | 处理建议 |
|---|---|
| 多个布尔字段表达互斥状态 | 改成判别联合 |
| 滥用交叉类型 | 确认同名字段不会产生不可用的 `never` |
| 索引签名返回值过宽 | 使用 `Record<K, V>` 或显式 key 联合 |
| 元组当普通数组传递 | 使用命名元组和 `readonly` 保留语义 |
| `interface` 和 `type` 随意混用 | 建立团队规则，公共对象 interface，组合表达 type |
