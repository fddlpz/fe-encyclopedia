## 背景与问题

TypeScript 是 JavaScript 的静态类型超集。它在编译期检查类型，但运行时仍然是 JavaScript；类型不会自动生成运行时校验，也不会改变浏览器或 Node.js 的执行语义。

在前端工程中，类型系统基础：静态类型 vs 动态类型、类型推断、类型断言、unknown vs any、never通常不是单独存在的知识点，而是连接语言特性、浏览器能力、框架抽象和工程约束的判断入口。理解它的背景，可以帮助我们回答三个问题：为什么需要这个机制、它解决了什么边界问题，以及在什么场景下不应该过度使用。

## 工作原理

| 概念 | 含义 | 工程意义 |
|---|---|---|
| 静态类型 | 运行前检查变量、参数、返回值和对象结构 | 提前发现契约错误 |
| 动态类型 | 运行时值才决定实际行为 | 仍要处理 JSON、表单、URL 等外部输入 |
| 类型推断 | 编译器根据赋值、返回值和控制流推导类型 | 少写冗余类型，保留可读性 |
| 类型断言 | 告诉编译器“我比你更清楚” | 应用于边界适配，不应掩盖错误 |
| `unknown` | 安全的未知类型，使用前必须收窄 | 适合外部输入 |
| `any` | 关闭类型检查 | 适合迁移过渡，不能长期扩散 |
| `never` | 不可能出现的值 | 用于穷尽性检查和不可达状态 |

TypeScript 的核心价值不是“写更多类型”，而是把业务不变量放到编译期。比如订单状态、接口返回、组件 props、路由参数和权限模型都应该有明确类型边界。

## 代码示例

外部输入优先用 `unknown`，再通过类型守卫收窄：

```ts
type UserProfile = {
  id: string
  name: string
}

function isUserProfile(value: unknown): value is UserProfile {
  if (typeof value !== "object" || value === null) {
    return false
  }

  const record = value as Record<string, unknown>

  // 中文注释：运行时校验外部数据，类型守卫通过后 TypeScript 才信任结构。
  return typeof record.id === "string" && typeof record.name === "string"
}

function parseUserProfile(raw: unknown) {
  if (!isUserProfile(raw)) {
    throw new Error("Invalid user profile")
  }

  return raw
}
```

`never` 适合做联合类型穷尽检查：

```ts
type LoadingState =
  | { status: "idle" }
  | { status: "loading" }
  | { status: "success"; data: string[] }
  | { status: "error"; message: string }

function renderState(state: LoadingState) {
  switch (state.status) {
    case "idle":
      return "等待加载"
    case "loading":
      return "加载中"
    case "success":
      return state.data.join(", ")
    case "error":
      return state.message
    default: {
      // 中文注释：新增状态但忘记处理时，这里会触发编译错误。
      const exhaustiveCheck: never = state
      return exhaustiveCheck
    }
  }
}
```

类型断言应尽量靠近边界：

```ts
const root = document.getElementById("app")

if (!(root instanceof HTMLElement)) {
  throw new Error("Missing #app root")
}

// 中文注释：经过运行时判断后，后续代码不需要再使用强制断言。
root.dataset.ready = "true"
```

## 工程实践

团队落地 TypeScript 时，应先启用 `strict`，再根据项目成熟度逐步打开更严格选项，例如 `noUncheckedIndexedAccess`、`exactOptionalPropertyTypes`、`noImplicitOverride`。这些选项会暴露更多边界问题，但能减少线上空值和索引访问错误。

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true
  }
}
```

类型设计应遵循三个原则：

1. 边界显式：接口响应、持久化数据、消息事件和环境变量必须先校验再进入业务层。
2. 内部简洁：函数内部依赖推断，不要给每个局部变量重复标注类型。
3. 不变量优先：用联合类型、字面量类型和 `never` 表达“哪些状态不可能同时存在”。

## 常见误区

| 常见问题 | 处理建议 |
|---|---|
| 用 `any` 快速通过编译 | 改用 `unknown`，在边界写类型守卫或 schema 校验 |
| 断言过多 | 先补运行时判断，再让控制流自动收窄 |
| 以为 TS 会校验 JSON | TS 类型只存在编译期，外部输入仍要运行时校验 |
| 手写冗余类型 | 能稳定推断时交给编译器，公共 API 再显式标注 |
| 忽略 `never` | 复杂状态机用 `never` 做穷尽性保护 |
