## 代码示例

下面的示例用一个轻量检查清单描述如何把「暗黑模式与主题切换」落到工程评审中：

```ts
interface ArticleCheckpoint {
  /** 检查项名称。 */
  title: string
  /** 检查项需要满足的要求。 */
  requirement: string
  /** 当前检查项是否已经完成。 */
  done: boolean
}

/**
 * 筛选仍未完成的检查项。
 * @param checkpoints 当前主题的工程检查清单。
 * @returns 尚未完成的检查项列表。
 */
function getUnfinishedCheckpoints(checkpoints: ArticleCheckpoint[]): ArticleCheckpoint[] {
  return checkpoints.filter((checkpoint) => {
    // 中文注释：只保留未完成项，便于在评审或上线前集中处理风险。
    return !checkpoint.done
  })
}

const checkpoints: ArticleCheckpoint[] = [
  {
    title: "概念边界",
    requirement: "已经说明适用场景、限制条件和替代方案。",
    done: true,
  },
  {
    title: "工程验证",
    requirement: "已经准备测试、日志或性能指标来验证结论。",
    done: false,
  },
]

console.table(getUnfinishedCheckpoints(checkpoints))
```

## 工程实践

暗黑模式与主题切换 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| token 层级 | 派生状态优先 computed，副作用用 watch，跨页面状态再引入 store。 | 用 Vue DevTools 检查依赖、更新次数、组件边界和 hydration mismatch。 |
| 组件 API | 先沉淀 token 命名、组件状态和可访问性，再扩大组件覆盖范围。 | 通过视觉回归、a11y、token 覆盖率和破坏性变更记录验证。 |
| 主题模式 | 在“设计稿到代码”场景下判断收益、成本、失败路径和回退策略。 | token 覆盖率 |
| 图标与资源 | 在“品牌一致性治理”场景下判断收益、成本、失败路径和回退策略。 | 组件使用分布 |

在真实项目中落地暗黑模式与主题切换时，可以按以下顺序推进：

| 步骤 | 动作 | 验证方式 |
|---|---|---|
| 识别场景 | 明确它解决的是性能、可维护性、兼容性、安全性还是协作问题。 | 写出触发条件和不适用场景。 |
| 建立基线 | 记录当前实现、指标、错误率或维护成本。 | 使用测试、日志、监控或构建报告作为证据。 |
| 小步改造 | 先在低风险路径验证方案，再扩大到核心链路。 | 对比改造前后的行为和指标。 |
| 固化规范 | 将经验沉淀为 lint、测试、文档或评审清单。 | 新增场景能复用同一套判断标准。 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 深度 watch 大对象会让性能和可预测性同时变差。 | 先用最小场景复现，再决定是否升级方案。 |
| 只按页面复制组件，会让组件库缺少稳定 API 和可复用语义。 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| 只追求视觉还原 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| token 命名缺少语义 | 先用最小场景复现，再决定是否升级方案。 |
| 没有版本策略和废弃流程 | 先用最小场景复现，再决定是否升级方案。 |
