## 章节定位

图标系统 不再只按“是什么、怎么用”展开，而是放回「第 19 章 设计系统与 UI 工程」的知识链路中理解：把设计语言、token、组件 API、主题、响应式和治理流程连接成可持续交付系统。

> **本节要解决的真实问题**：当项目里遇到 图标系统、图标、Icon、SVG 相关设计、选型、接入或排障时，能说清楚边界、取舍、验证方式和失败后的回退策略。

### 学习目标

| 层次 | 学完后应该能做到 |
|---|---|
| 概念定位 | 解释 图标系统 在 设计系统与 UI 工程 中解决的真实问题 |
| 机制理解 | 拆解 图标、Icon 与失败路径 |
| 工程判断 | 为 SVG 选择可验证、可回退的落地方案 |
| 验证闭环 | 围绕“token 是否有语义”建立判断标准 |

## 知识地图

设计系统不是组件仓库，而是让设计约束、代码实现和版本治理共享同一套语义。

| 层级 | 本节关注点 | 判断证据 |
|---|---|---|
| 问题入口 | 图标系统、图标、Icon 会在什么业务场景中暴露 | 组件库建设、多主题产品 |
| 运行机制 | token 是否有语义；组件状态是否完整；无障碍是否内建 | 视觉回归、a11y 检查 |
| 工程约束 | token 层级、组件 API、主题模式、图标与资源 | 兼容性、性能、安全、维护成本和回退方案 |
| 关联知识 | 与同章和跨章内容形成可追踪链路 | 关系链接、示例代码、测试和线上数据 |

## 核心机制

这一节建议沿着“术语 -> 机制 -> 证据”的顺序阅读。先不要急着背 API，先确认每个概念在系统里承担什么职责，以及它失败时会出现什么现象。

| 主题 | 需要理解的机制 | 可观察证据 |
|---|---|---|
| 图标系统 | 设计系统把视觉语义、组件 API、主题变量和治理流程绑定到同一套约束里。 | 通过视觉回归、a11y、token 覆盖率和破坏性变更记录验证。 |
| 图标 | 图形与媒体链路把数据、几何、纹理、像素、音视频帧和交互事件映射到设备能力上。 | 用 FPS、内存、首帧、卡顿率、DPR 和事件命中数据验证。 |
| Icon | 围绕“无障碍是否内建”拆解概念边界、运行机制和失败路径。 | token 覆盖率 |
| SVG | 围绕“变更如何发布”拆解概念边界、运行机制和失败路径。 | 组件使用分布 |

<Callout type="tip" title="阅读方法">
读这一节时，建议为每个关键词补一个“最小复现场景”和一个“线上验证信号”。能被复现和验证的知识，才真正能进入工程判断。
</Callout>

## 工程决策

图标系统 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| token 层级 | 先沉淀 token 命名、组件状态和可访问性，再扩大组件覆盖范围。 | 通过视觉回归、a11y、token 覆盖率和破坏性变更记录验证。 |
| 组件 API | 按数据量、更新频率、交互复杂度、导出需求和可访问性选择技术栈。 | 用 FPS、内存、首帧、卡顿率、DPR 和事件命中数据验证。 |
| 主题模式 | 在“设计稿到代码”场景下判断收益、成本、失败路径和回退策略。 | token 覆盖率 |
| 图标与资源 | 在“品牌一致性治理”场景下判断收益、成本、失败路径和回退策略。 | 组件使用分布 |

## 实战路径

推荐按四步推进：

1. **界定场景**：明确当前是新功能接入、旧系统治理、性能优化、安全加固还是线上排障。
2. **建立样例**：用最小代码或配置复现 图标系统 的核心行为，不直接在完整业务里猜测。
3. **接入约束**：把 token 层级、组件 API、主题模式 写进方案，说明不满足时如何降级。
4. **验证沉淀**：用 视觉回归、a11y 检查、token 覆盖率 形成证据，并把结论回写到团队文档。

```ts
type ChapterDecision = {
  objective: string
  constraints: string[]
  signals: string[]
  rollback: string
}

/**
 * 生成本节主题的落地决策记录。
 * @param owner 负责验证和维护该方案的角色或小组。
 * @returns 可写入技术方案或评审记录的决策对象。
 */
function createDecisionRecord(owner: string): ChapterDecision {
  return {
    objective: "图标系统",
    constraints: ["图标系统","图标","Icon","SVG"],
    signals: ["视觉回归","a11y 检查","token 覆盖率"],
    // 中文注释：回退策略必须在接入前定义，避免上线后只能被动救火。
    rollback: `由 ${owner} 负责准备降级开关、兼容方案和验证清单`,
  }
}

console.table(createDecisionRecord("frontend"))
```

## 诊断与验证

| 现象 | 定位动作 | 判断完成的证据 |
|---|---|---|
| 行为与预期不一致 | 回到最小样例，逐步加入真实数据、环境和配置 | 能明确指出差异来自输入、运行时、配置还是依赖 |
| 性能或稳定性退化 | 记录基线，再观察 视觉回归、a11y 检查、token 覆盖率 | 指标回到预算内，并且关键路径有回归测试 |
| 团队理解不一致 | 把术语、边界、反例和回退策略写进评审记录 | 新成员可以按同一清单复现和验证 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 只按页面复制组件，会让组件库缺少稳定 API 和可复用语义。 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| 只看视觉效果而不测低端设备，会把渲染成本留给用户。 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| 只追求视觉还原 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| token 命名缺少语义 | 先用最小场景复现，再决定是否升级方案。 |
| 没有版本策略和废弃流程 | 先用最小场景复现，再决定是否升级方案。 |
