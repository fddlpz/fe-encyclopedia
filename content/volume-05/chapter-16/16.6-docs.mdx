## 背景与问题

文档与注释 不再只按“是什么、怎么用”展开，而是放回「第 16 章 代码质量与规范」的知识链路中理解：把格式化、静态检查、类型检查、提交门禁、文档和评审串成前移反馈闭环。

> **本节要解决的真实问题**：当项目里遇到 文档与注释、文档、注释、JSDoc 相关设计、选型、接入或排障时，能说清楚边界、取舍、验证方式和失败后的回退策略。

## 核心概念

质量工具不是为了制造规则，而是把机器能判断的问题交给机器，把代码审查留给设计与风险。

| 层级 | 本节关注点 | 判断证据 |
|---|---|---|
| 问题入口 | 文档与注释、文档、注释 会在什么业务场景中暴露 | 团队协作规范、遗留项目治理 |
| 运行机制 | 规则能否自动修复；CI 是否强制执行；团队是否理解规则原因 | lint 失败分布、typecheck 耗时 |
| 工程约束 | 规则严格度、本地反馈速度、CI 门禁、文档约定 | 兼容性、性能、安全、维护成本和回退方案 |
| 关联知识 | 与同章和跨章内容形成可追踪链路 | 关系链接、示例代码、测试和线上数据 |

## 代码示例

下面的示例用一个轻量检查清单描述如何把「文档与注释」落到工程评审中：

```ts
interface ArticleCheckpoint {
  /** 检查项名称。 */
  title: string
  /** 检查项需要满足的要求。 */
  requirement: string
  /** 当前检查项是否已经完成。 */
  done: boolean
}

/**
 * 筛选仍未完成的检查项。
 * @param checkpoints 当前主题的工程检查清单。
 * @returns 尚未完成的检查项列表。
 */
function getUnfinishedCheckpoints(checkpoints: ArticleCheckpoint[]): ArticleCheckpoint[] {
  return checkpoints.filter((checkpoint) => {
    // 中文注释：只保留未完成项，便于在评审或上线前集中处理风险。
    return !checkpoint.done
  })
}

const checkpoints: ArticleCheckpoint[] = [
  {
    title: "概念边界",
    requirement: "已经说明适用场景、限制条件和替代方案。",
    done: true,
  },
  {
    title: "工程验证",
    requirement: "已经准备测试、日志或性能指标来验证结论。",
    done: false,
  },
]

console.table(getUnfinishedCheckpoints(checkpoints))
```

## 工程实践

文档与注释 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| 规则严格度 | 能自动修复的交给工具，涉及设计、边界和风险的留给代码审查。 | 统计规则命中、自动修复比例、CI 失败原因和评审返工原因。 |
| 本地反馈速度 | 在“遗留项目治理”场景下判断收益、成本、失败路径和回退策略。 | typecheck 耗时 |
| CI 门禁 | 在“组件库维护”场景下判断收益、成本、失败路径和回退策略。 | pre-commit 命中率 |
| 文档约定 | 在“大型仓库门禁”场景下判断收益、成本、失败路径和回退策略。 | 代码审查返工原因 |

在真实项目中落地文档与注释时，可以按以下顺序推进：

| 步骤 | 动作 | 验证方式 |
|---|---|---|
| 识别场景 | 明确它解决的是性能、可维护性、兼容性、安全性还是协作问题。 | 写出触发条件和不适用场景。 |
| 建立基线 | 记录当前实现、指标、错误率或维护成本。 | 使用测试、日志、监控或构建报告作为证据。 |
| 小步改造 | 先在低风险路径验证方案，再扩大到核心链路。 | 对比改造前后的行为和指标。 |
| 固化规范 | 将经验沉淀为 lint、测试、文档或评审清单。 | 新增场景能复用同一套判断标准。 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 规则没有解释和自动修复能力时，团队很容易绕过工具。 | 先用最小场景复现，再决定是否升级方案。 |
| 规则过严导致绕过工具 | 先用最小场景复现，再决定是否升级方案。 |
| 只在本地执行 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| 把风格争论留到代码审查 | 先用最小场景复现，再决定是否升级方案。 |

## 对比总结

| 维度 | 关注点 | 判断标准 |
|---|---|---|
| 概念理解 | 文档与注释的定义、边界和相邻概念。 | 能用自己的话解释，并指出至少一个反例。 |
| 工程使用 | 代码、配置、流程或团队规范中的落地方式。 | 能在真实场景中给出可执行步骤。 |
| 风险控制 | 性能、兼容、安全、可维护性和协作成本。 | 有测试、监控、回滚或审查机制支撑。 |
| 长期维护 | 文档、自动化检查和知识传递。 | 后续成员能按同一标准继续演进。 |
