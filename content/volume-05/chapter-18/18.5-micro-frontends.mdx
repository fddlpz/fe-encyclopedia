## 核心概念

架构不是目录漂亮，而是让需求变化能被限制在局部，让团队知道代码应该放在哪里、依赖谁、如何发布。

| 层级 | 本节关注点 | 判断证据 |
|---|---|---|
| 问题入口 | 微前端架构、iframe、Web 会在什么业务场景中暴露 | 大型中后台、多团队协作 |
| 运行机制 | 领域边界在哪里；依赖方向是否清晰；状态是否可追踪 | 循环依赖、公共层膨胀 |
| 工程约束 | 分层方式、组件粒度、状态模型、微前端边界 | 兼容性、性能、安全、维护成本和回退方案 |
| 关联知识 | 与同章和跨章内容形成可追踪链路 | 关系链接、示例代码、测试和线上数据 |

下面的示例用一个轻量检查清单描述如何把「微前端架构：iframe / Web Components / Module Federation / qiankun」落到工程评审中：

```ts
interface ArticleCheckpoint {
  /** 检查项名称。 */
  title: string
  /** 检查项需要满足的要求。 */
  requirement: string
  /** 当前检查项是否已经完成。 */
  done: boolean
}

/**
 * 筛选仍未完成的检查项。
 * @param checkpoints 当前主题的工程检查清单。
 * @returns 尚未完成的检查项列表。
 */
function getUnfinishedCheckpoints(checkpoints: ArticleCheckpoint[]): ArticleCheckpoint[] {
  return checkpoints.filter((checkpoint) => {
    // 中文注释：只保留未完成项，便于在评审或上线前集中处理风险。
    return !checkpoint.done
  })
}

const checkpoints: ArticleCheckpoint[] = [
  {
    title: "概念边界",
    requirement: "已经说明适用场景、限制条件和替代方案。",
    done: true,
  },
  {
    title: "工程验证",
    requirement: "已经准备测试、日志或性能指标来验证结论。",
    done: false,
  },
]

console.table(getUnfinishedCheckpoints(checkpoints))
```

## 工程实践

微前端架构：iframe / Web Components / Module Federation / qiankun 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| 分层方式 | 适合需要深度定制、复杂兼容或 Module Federation 的大型项目；简单应用要评估配置成本。 | 检查 chunk 拆分、sideEffects、缓存命中和 federation 远程模块加载失败路径。 |
| 组件粒度 | 只有当复杂度、团队边界或发布节奏真的需要时，才引入更重的架构方案。 | 观察跨模块改动范围、循环依赖、公共层膨胀和故障隔离效果。 |

| 微前端边界 | 在“渐进式重构”场景下判断收益、成本、失败路径和回退策略。 | 发布频率 |
