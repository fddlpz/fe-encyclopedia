## 工程决策

编译原理与前端工具链实现 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| parser 选择 | 在“Babel 插件”场景下判断收益、成本、失败路径和回退策略。 | AST Explorer |
| AST 兼容性 | 在“lint 规则”场景下判断收益、成本、失败路径和回退策略。 | 插件快照测试 |
| 转换顺序 | 在“代码生成”场景下判断收益、成本、失败路径和回退策略。 | source map 映射 |
| 性能 | 在“迁移脚本”场景下判断收益、成本、失败路径和回退策略。 | 构建耗时 |

## 实战路径

推荐按四步推进：

1. **界定场景**：明确当前是新功能接入、旧系统治理、性能优化、安全加固还是线上排障。
2. **建立样例**：用最小代码或配置复现 编译原理与前端工具链实现 的核心行为，不直接在完整业务里猜测。
3. **接入约束**：把 parser 选择、AST 兼容性、转换顺序 写进方案，说明不满足时如何降级。
4. **验证沉淀**：用 AST Explorer、插件快照测试、source map 映射 形成证据，并把结论回写到团队文档。

  objective: string
  constraints: string[]

/**

 * @param owner 负责验证和维护该方案的角色或小组。
 * @returns 可写入技术方案或评审记录的决策对象。
 */
function createDecisionRecord(owner: string): ChapterDecision {
  return {

## 诊断与验证

| 现象 | 定位动作 | 判断完成的证据 |
|---|---|---|
| 行为与预期不一致 | 回到最小样例，逐步加入真实数据、环境和配置 | 能明确指出差异来自输入、运行时、配置还是依赖 |
| 性能或稳定性退化 | 记录基线，再观察 AST Explorer、插件快照测试、source map 映射 | 指标回到预算内，并且关键路径有回归测试 |
| 团队理解不一致 | 把术语、边界、反例和回退策略写进评审记录 | 新成员可以按同一清单复现和验证 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 用正则改代码 | 先用最小场景复现，再决定是否升级方案。 |
| 忽略注释和格式保留 | 先用最小场景复现，再决定是否升级方案。 |
