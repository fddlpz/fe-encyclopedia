## 代码示例

下面的示例用一个轻量检查清单描述如何把「Contract Testing：OpenAPI / GraphQL Schema 契约校验」落到工程评审中：

```ts
interface ArticleCheckpoint {
  /** 检查项名称。 */
  title: string
  /** 检查项需要满足的要求。 */
  requirement: string
  /** 当前检查项是否已经完成。 */
  done: boolean
}

/**
 * 筛选仍未完成的检查项。
 * @param checkpoints 当前主题的工程检查清单。
 * @returns 尚未完成的检查项列表。
 */
function getUnfinishedCheckpoints(checkpoints: ArticleCheckpoint[]): ArticleCheckpoint[] {
  return checkpoints.filter((checkpoint) => {
    // 中文注释：只保留未完成项，便于在评审或上线前集中处理风险。
    return !checkpoint.done
  })
}

const checkpoints: ArticleCheckpoint[] = [
  {
    title: "概念边界",
    requirement: "已经说明适用场景、限制条件和替代方案。",
    done: true,
  },
  {
    title: "工程验证",
    requirement: "已经准备测试、日志或性能指标来验证结论。",
    done: false,
  },
]

console.table(getUnfinishedCheckpoints(checkpoints))
```

## 工程实践

Contract Testing：OpenAPI / GraphQL Schema 契约校验 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| 单元/集成/E2E | 高价值路径优先覆盖，低风险展示逻辑避免用昂贵 E2E 堆砌。 | 关注 flaky rate、失败定位速度、覆盖率变化和 CI 总时长。 |
| 测试数据 | 先定义对前端稳定的契约，再决定 REST、GraphQL、tRPC、BFF 或 Serverless 边界。 | 用契约测试、超时注入、压测、trace 和结构化日志证明服务可靠。 |
| Mock 策略 | 在“API 契约”场景下判断收益、成本、失败路径和回退策略。 | CI 失败日志 |
| 可访问性 | 在“性能回归”场景下判断收益、成本、失败路径和回退策略。 | 截图差异 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 只看覆盖率数字会掩盖断言质量和关键路径缺口。 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| 直接透传后端内部模型，会让前端契约随服务端实现一起震荡。 | 先用最小场景复现，再决定是否升级方案。 |
| 只追求覆盖率 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| E2E 过多导致反馈慢 | 先用最小场景复现，再决定是否升级方案。 |
| 测试依赖执行顺序或真实外部服务 | 先用最小场景复现，再决定是否升级方案。 |
