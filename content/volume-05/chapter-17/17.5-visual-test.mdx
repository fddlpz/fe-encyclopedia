## 章节定位

可视化测试：Storybook / Chromatic 不再只按“是什么、怎么用”展开，而是放回「第 17 章 测试策略」的知识链路中理解：按风险选择测试层级，用自动化证据证明关键行为、交互路径和跨模块契约不会退化。

> **本节要解决的真实问题**：当项目里遇到 可视化测试、Storybook、Chromatic、视觉回归 相关设计、选型、接入或排障时，能说清楚边界、取舍、验证方式和失败后的回退策略。

### 学习目标

| 层次 | 学完后应该能做到 |
|---|---|
| 概念定位 | 解释 可视化测试 在 测试体系 中解决的真实问题 |
| 机制理解 | 拆解 Storybook、Chromatic 与失败路径 |
| 工程判断 | 为 视觉回归 选择可验证、可回退的落地方案 |
| 验证闭环 | 围绕“风险在哪一层”建立判断标准 |

## 知识地图

测试不是覆盖率数字，而是对业务行为的可执行描述；不同层级测试负责不同风险。

| 层级 | 本节关注点 | 判断证据 |
|---|---|---|
| 问题入口 | 可视化测试、Storybook、Chromatic 会在什么业务场景中暴露 | 核心业务流程、组件交互 |
| 运行机制 | 风险在哪一层；测试是否稳定；失败能否定位 | flaky rate、覆盖率差异 |
| 工程约束 | 单元/集成/E2E、测试数据、Mock 策略、可访问性 | 兼容性、性能、安全、维护成本和回退方案 |
| 关联知识 | 与同章和跨章内容形成可追踪链路 | 关系链接、示例代码、测试和线上数据 |

## 核心机制

这一节建议沿着“术语 -> 机制 -> 证据”的顺序阅读。先不要急着背 API，先确认每个概念在系统里承担什么职责，以及它失败时会出现什么现象。

| 主题 | 需要理解的机制 | 可观察证据 |
|---|---|---|
| 可视化测试 | 质量工具把格式、静态规则、类型和协作约定变成可重复执行的反馈。 | 统计规则命中、自动修复比例、CI 失败原因和评审返工原因。 |
| Storybook | 测试层级应与风险层级匹配：单元验证逻辑，组件验证交互，E2E 验证关键路径，契约验证协作边界。 | 关注 flaky rate、失败定位速度、覆盖率变化和 CI 总时长。 |
| Chromatic | 围绕“失败能否定位”拆解概念边界、运行机制和失败路径。 | CI 失败日志 |
| 视觉回归 | 围绕“Mock 边界是否真实”拆解概念边界、运行机制和失败路径。 | 截图差异 |

## 工程决策

可视化测试：Storybook / Chromatic 的价值在于帮助你做判断，而不是增加术语库存。遇到真实项目时，可以按下面的决策表检查。

| 决策维度 | 应该如何判断 | 验证方式 |
|---|---|---|
| 单元/集成/E2E | 能自动修复的交给工具，涉及设计、边界和风险的留给代码审查。 | 统计规则命中、自动修复比例、CI 失败原因和评审返工原因。 |
| 测试数据 | 高价值路径优先覆盖，低风险展示逻辑避免用昂贵 E2E 堆砌。 | 关注 flaky rate、失败定位速度、覆盖率变化和 CI 总时长。 |
| Mock 策略 | 在“API 契约”场景下判断收益、成本、失败路径和回退策略。 | CI 失败日志 |
| 可访问性 | 在“性能回归”场景下判断收益、成本、失败路径和回退策略。 | 截图差异 |

## 实战路径

推荐按四步推进：

1. **界定场景**：明确当前是新功能接入、旧系统治理、性能优化、安全加固还是线上排障。
2. **建立样例**：用最小代码或配置复现 可视化测试 的核心行为，不直接在完整业务里猜测。
3. **接入约束**：把 单元/集成/E2E、测试数据、Mock 策略 写进方案，说明不满足时如何降级。
4. **验证沉淀**：用 flaky rate、覆盖率差异、CI 失败日志 形成证据，并把结论回写到团队文档。

```ts
type ChapterDecision = {
  objective: string
  constraints: string[]
  signals: string[]
  rollback: string
}

/**
 * 生成本节主题的落地决策记录。
 * @param owner 负责验证和维护该方案的角色或小组。
 * @returns 可写入技术方案或评审记录的决策对象。
 */
function createDecisionRecord(owner: string): ChapterDecision {
  return {
    objective: "Storybook / Chromatic",
    constraints: ["可视化测试","Storybook","Chromatic","视觉回归"],
    signals: ["flaky rate","覆盖率差异","CI 失败日志"],
    // 中文注释：回退策略必须在接入前定义，避免上线后只能被动救火。
    rollback: `由 ${owner} 负责准备降级开关、兼容方案和验证清单`,
  }
}

console.table(createDecisionRecord("frontend"))
```

## 诊断与验证

| 现象 | 定位动作 | 判断完成的证据 |
|---|---|---|
| 行为与预期不一致 | 回到最小样例，逐步加入真实数据、环境和配置 | 能明确指出差异来自输入、运行时、配置还是依赖 |
| 性能或稳定性退化 | 记录基线，再观察 flaky rate、覆盖率差异、CI 失败日志 | 指标回到预算内，并且关键路径有回归测试 |
| 团队理解不一致 | 把术语、边界、反例和回退策略写进评审记录 | 新成员可以按同一清单复现和验证 |

## 常见误区

| 误区 | 更可靠的处理方式 |
|---|---|
| 规则没有解释和自动修复能力时，团队很容易绕过工具。 | 先用最小场景复现，再决定是否升级方案。 |
| 只看覆盖率数字会掩盖断言质量和关键路径缺口。 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| 只追求覆盖率 | 把问题拆到上下文、边界、证据和回退方案中验证。 |
| E2E 过多导致反馈慢 | 先用最小场景复现，再决定是否升级方案。 |
| 测试依赖执行顺序或真实外部服务 | 先用最小场景复现，再决定是否升级方案。 |
