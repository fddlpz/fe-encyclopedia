## 背景与问题

Performance Timeline 是浏览器暴露性能条目的统一时间轴。PerformanceObserver 允许页面实时订阅导航、资源、绘制、布局偏移、长任务等事件。它是 RUM（真实用户监控）的基础。

在前端工程中，性能与观测 API：Performance Timeline、PerformanceObserver、Scheduler API通常不是单独存在的知识点，而是连接语言特性、浏览器能力、框架抽象和工程约束的判断入口。理解它的背景，可以帮助我们回答三个问题：为什么需要这个机制、它解决了什么边界问题，以及在什么场景下不应该过度使用。

## 工作原理

| Entry 类型 | 用途 |
|---|---|
| `navigation` | HTML 文档导航耗时 |
| `resource` | CSS、JS、图片、fetch 等资源耗时 |
| `paint` | FP、FCP |
| `largest-contentful-paint` | LCP |
| `layout-shift` | CLS |
| `event` | 交互事件耗时 |
| `longtask` | 主线程 50ms 以上任务 |

Scheduler API 仍需按兼容性渐进使用。即使没有 Scheduler，也可以用 `requestIdleCallback`、`requestAnimationFrame` 和任务切分降低阻塞。

## 代码示例

订阅资源耗时：

```ts
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntriesByType("resource")) {
    const resource = entry as PerformanceResourceTiming

    // 中文注释：跨源资源需要 Timing-Allow-Origin 才能看到完整细节。
    console.log(resource.name, resource.duration)
  }
})

observer.observe({ type: "resource", buffered: true })
```

监听长任务：

```ts
new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    // 中文注释：长任务超过 50ms，通常会影响输入响应。
    console.warn("long task", entry.duration)
  }
}).observe({ type: "longtask", buffered: true })
```

自定义标记：

```ts
performance.mark("article-render-start")

// 中文注释：业务渲染完成后记录可观测的自定义耗时。
performance.mark("article-render-end")
performance.measure("article-render", "article-render-start", "article-render-end")
```

## 工程实践

生产监控建议：

1. 核心 Web Vitals：LCP、CLS、INP。
2. 导航和资源：TTFB、关键 CSS/JS/图片耗时。
3. 主线程：Long Tasks、hydration、第三方脚本。
4. 业务指标：首个可用按钮、列表可交互、编辑器 ready。

上报要采样、去重、脱敏，并通过 `sendBeacon` 或 `fetch(..., { keepalive: true })` 在页面隐藏时发送。

## 常见误区

| 常见问题 | 处理建议 |
|---|---|
| 只用实验室 Lighthouse | 同时采集真实用户 RUM |
| 跨源资源 timing 不完整 | 服务端设置 `Timing-Allow-Origin` |
| 指标无上下文 | 关联设备、网络、页面、版本和实验分组 |
| 上报影响性能 | 采样、批量、低优先级发送 |
| 自定义指标没有定义 | 明确开始/结束点和业务含义 |
