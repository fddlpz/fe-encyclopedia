## 背景与问题

经典脚本默认会阻塞 HTML 解析：浏览器必须下载并执行脚本，才能继续解析后续文档。预加载扫描器会在主解析器阻塞时尽量发现后面的资源，但它不能理解所有动态生成的依赖。

在前端工程中，脚本执行与解析阻塞：async/defer/module、预加载扫描器、preload/prefetch通常不是单独存在的知识点，而是连接语言特性、浏览器能力、框架抽象和工程约束的判断入口。理解它的背景，可以帮助我们回答三个问题：为什么需要这个机制、它解决了什么边界问题，以及在什么场景下不应该过度使用。

## 工作原理

| 脚本形式 | 下载 | 执行 | 顺序 |
|---|---|---|---|
| 普通脚本 | 阻塞解析 | 下载后立即执行 | 文档顺序 |
| `defer` | 不阻塞解析 | DOM 解析完成后执行 | 文档顺序 |
| `async` | 不阻塞解析 | 下载完成后尽快执行 | 不保证顺序 |
| `type="module"` | 默认延迟执行 | 支持 import 依赖图 | 模块依赖顺序 |

根据 MDN，`defer` 对 module scripts 没有效果，因为模块脚本默认就是延迟执行。

## 代码示例

应用主脚本使用 defer 或 module：

```html
<!-- 中文注释：主应用脚本通常依赖 DOM，使用 defer 保证顺序且不阻塞解析。 -->
<script src="/assets/app.js" defer></script>

<!-- 中文注释：现代应用可使用模块脚本，依赖由浏览器按模块图解析。 -->
<script type="module" src="/assets/app.mjs"></script>
```

第三方独立脚本使用 async：

```html
<!-- 中文注释：统计脚本不应阻塞页面主流程，也不依赖其他脚本顺序。 -->
<script src="https://analytics.example.com/sdk.js" async></script>
```

资源提示：

```html
<link rel="preload" href="/assets/app.css" as="style" />
<link rel="prefetch" href="/next-page-data.json" as="fetch" crossorigin />

<!-- 中文注释：importmap 必须在依赖它的 module script 前声明。 -->
<script type="importmap">
{
  "imports": {
    "date-fns": "/vendor/date-fns/index.js"
  }
}
</script>
```

## 工程实践

脚本加载策略：

1. 首屏必要脚本：尽量小，使用 defer/module，避免阻塞解析。
2. 第三方脚本：async、延迟加载、权限隔离和失败兜底。
3. 路由后续资源：prefetch 只能作为低优先级提示，不保证一定加载。
4. 关键 CSS/字体：preload 要谨慎，避免抢占主资源。

```ts
async function loadFeature() {
  const module = await import("./heavy-feature.js")

  // 中文注释：交互触发后再加载重型能力，减少首屏 JS。
  module.mount()
}
```

## 常见误区

| 常见问题 | 处理建议 |
|---|---|
| async 脚本依赖执行顺序 | 有顺序依赖用 defer 或模块依赖 |
| preload 过多 | 只预加载首屏关键资源 |
| module 忘记兼容目标 | 结合目标浏览器和构建输出验证 |
| 第三方脚本阻塞主线程 | 延迟加载、隔离、监控 Long Task |
| 动态插入脚本缺少错误处理 | 监听 `onerror`，提供降级路径 |
