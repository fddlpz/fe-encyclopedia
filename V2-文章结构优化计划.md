# 前端开发百科全书 V2 文章结构优化计划

> 生成日期：2026-06-26  
> 适用范围：`content/` 下全部 MDX 文章，以及文章读取、搜索索引、导航和质量检查相关脚本。

## 1. 计划背景

当前百科全书已经完成大规模内容补写和框架重构，站点共包含 231 篇 MDX 文章。现有内容已经具备「原理、用法、实践、陷阱、参考」等基础结构，但不同批次文章在元数据、章节深度、正文标题、参考资料格式和质量验收维度上仍不完全一致。

V2 优化的目标不是重新写一遍百科，而是把现有内容升级成一套可持续维护的内容工程体系：每篇文章都有稳定的数据结构、稳定的正文骨架、可验证的质量标准，以及可被搜索、推荐、知识图谱和后续自动化脚本复用的元信息。

## 3. V2 总体目标

1. 统一文章 frontmatter 字段，让页面、搜索、导航、知识图谱和内容检查脚本使用同一套数据契约。
2. 统一正文结构，让所有文章都从「简单定义型」升级为「可学习、可追溯、可实践」的专题文章。
3. 建立内容质量检查脚本，避免后续新增文章重新出现字段缺失、标题不统一、引用不可追溯等问题。
4. 对内容深度进行分层优化，优先保证基础学习路径完整，再推进框架、工程化、安全、全栈和跨端专题。
5. 让 `框架重构` 状态文章逐步转为 `已完成`，并把状态变化纳入可追踪记录。

## 6. 标准正文结构

所有文章正文建议统一为以下二级标题：

```mdx
## 速览

## 背景与问题

## 核心概念

## 工作原理

## 代码示例

## 工程实践

## 常见误区

## 对比总结

## 自测题

## 参考资料
```

### 6.1 各段落写作标准

| 模块 | 内容标准 |
|---|---|
| 速览 | 用 3 到 5 条写清本文核心结论、使用边界和工程判断。 |
| 背景与问题 | 说明概念为什么出现，解决什么真实问题，不只给定义。 |
| 核心概念 | 拆解关键术语、概念边界、相邻概念和必要前置知识。 |
| 工作原理 | 解释机制、流程、调度顺序、数据结构或设计取舍。 |
| 代码示例 | 至少包含基础示例和工程示例；高风险主题需补错误示例。 |
| 工程实践 | 给出真实业务场景、落地步骤、选型建议、验证方式和回退方案。 |
| 常见误区 | 写清错误表现、原因、修正方式，优先使用表格。 |
| 对比总结 | 与相似概念、方案或 API 做横向对比。 |
| 自测题 | 至少提供 1 个 `Quiz` 组件，用于检查关键理解。 |
| 参考资料 | 优先引用官方标准、官方文档、规范说明和权威工程实践。 |

## 7. 旧结构到 V2 结构的迁移关系

| 当前结构 | V2 目标结构 | 迁移说明 |
|---|---|---|
| `## 原理` | `## 背景与问题`、`## 核心概念`、`## 工作原理` | 将大段原理拆成问题背景、术语解释和机制说明。 |
| `## 用法` | `## 代码示例` | 保留 API、配置、语法示例，并补充运行结果或适用边界。 |
| `## 实践` | `## 工程实践` | 从简单案例升级为场景、约束、步骤、验证和回退方案。 |
| `## 陷阱` | `## 常见误区` | 保留表格形式，补充错误原因和修正方式。 |
| `## 参考` | `## 参考资料` | 统一为标题加链接格式，并同步到 frontmatter 的 `references`。 |
| 文章开头摘要 | `summary` 和 `## 速览` | 元数据用于索引和卡片，正文速览用于读者快速判断。 |

## 10. 建议新增质量检查脚本

建议新增 `scripts/check-content-structure.ts`，用于在提交前检查：

1. frontmatter 必填字段是否完整。
2. `status` 是否属于固定枚举。
3. `difficulty` 是否属于固定枚举。
4. 是否包含 V2 标准二级标题。
5. 是否至少包含 1 个代码块。
6. 是否至少包含 1 个 `Quiz`。
7. 是否至少包含 1 个参考资料。
8. `prerequisites`、`related`、`crossRefs` 中的 slug 是否存在。
9. `references` 是否是 `{ title, url }` 结构。
10. 是否存在空段落、模板残留或无效链接。

建议在 `package.json` 中增加：

```json
{
  "scripts": {
    "check-content": "tsx scripts/check-content-structure.ts"
  }
}
```

## 11. 单篇文章完成标准

| 检查项 | 标准 |
|---|---|
| 元数据 | 必填字段完整，状态、难度、引用格式符合类型约束。 |
| 速览 | 能在 30 秒内帮助读者判断文章价值和适用范围。 |
| 深度 | 不止解释是什么，还解释为什么、如何工作、如何验证。 |
| 示例 | 代码具备中文注释，并说明输入、输出或适用条件。 |
| 实践 | 包含真实工程场景、落地步骤和验证方式。 |
| 误区 | 至少覆盖 3 个常见错误或反模式。 |
| 对比 | 对相邻概念或方案给出表格化差异。 |
| 自测 | 至少 1 个选择题，并给出解释。 |
| 参考 | 优先官方文档和标准规范，避免只有泛泛链接。 |
| 关联 | 前置依赖、相关推荐和交叉引用能形成学习路径。 |

## 12. 章节完成标准

1. 章内所有小节均达到单篇文章完成标准。
2. 章内文章难度递进合理，基础、进阶、高级边界清晰。
3. 章内小节顺序符合真实学习路径。
4. 章内至少建立 3 条有效知识关联。
5. 章内所有示例、表格、外链和 MDX 组件通过构建验证。
6. 完成后同步更新 `优化计划.md` 的章节状态和补充记录。

## 13. GitHub 提交流程建议

每批内容优化建议按以下节奏提交：

1. 结构和脚本变更单独提交。
2. 每 1 到 2 个章节作为一个内容提交。
3. 大规模自动迁移提交与人工精修提交分开。
4. 每次推送前运行：

```bash
npm run generate-index
npm run build
```

推荐提交信息示例：

```bash
docs: add v2 article structure plan
content: migrate browser runtime articles to v2 structure
chore: add content structure checker
```
